How to Read These Blueprints

This manual answers one question for five giant systems: a piece of data is born at the source — what happens to it, step by step, until a human or model uses it, and what technology touches it at each hop?

What each blueprint contains

Every mega-project is documented as a flow, not a process. For each you get: the scale (the numbers that force the design), the source systems and how you connect to each, the end-to-end flow diagram, then a stage-by-stage walk where each hop states what tech, what it receives, what it does, what it emits, and why that tech. The heart of each is the transformation chain — numbered steps showing input → operation → output so you can see exactly what changes between raw and usable. Finally the full tech stack table and the failure/scale notes.

Read it like a data particle: pick one event (a click, a trip ping, a payment) and follow it top to bottom. If you can narrate that journey for even two of these platforms, you can hold your own with staff-level engineers.

Why "big" changes the design

At small scale you can do everything in one Spark job or a warehouse. At millions of events per second and petabytes, physics forces a specific shape: you must decouple producers from consumers (a durable log), separate the hot real-time path from the heavy batch path (they have different latency budgets), pre-compute anything a user-facing system needs (nobody scans a petabyte in a 50ms API call), and partition and pre-aggregate relentlessly because the cost of moving and scanning data dominates everything. All five blueprints are variations on that response.

Every project below is these 12 operations, in a specific order, split across a hot (streaming) and cold (batch) path, with a specific tech at each hop. Once you see that, the giants stop being intimidating.

The three paths you'll see repeatedly

Hot path (seconds): source → log → stream processor → online store/serving. Powers live features, fraud, tracking, personalization. Cannot touch the lakehouse in its latency budget.

Warm path (minutes): log → micro-batch → lakehouse silver/gold → near-real-time dashboards.

Cold path (hours): lakehouse → heavy batch → warehouse/OLAP + ML training → reporting/models. Where the petabyte-scale aggregation and feature history live.

This is the Lambda shape (parallel hot + cold, reconciled) or its simplified Kappa cousin (one streaming pipeline you replay from the log). Both appear below.

Stage-by-stage — what tech, what it does, why

① Collector service — receives SDK events over HTTPS, validates against the tracking plan, drops bot/spam, enriches with device/IP-geo, batches, and publishes to Kafka. Why here: a thin, massively-scalable edge that shields the log from garbage and does cheap enrichment before buffering.
② Kafka (the backbone) — durable, replayable, partitioned log; one topic per domain (clicks, orders, catalog…), keyed for ordering (by user/order id), governed by a Schema Registry. Why: decouples 1000s of producers/consumers, absorbs 10× peaks, and lets every downstream consumer read independently and replay history.
③ Flink (hot path) — stateful stream processing: sessionization, real-time counters/features, and event routing, all in the seconds budget. Why Flink over Spark here: true low-latency event-at-a-time processing with rich state and event-time windows; the outputs feed online serving where milliseconds matter.
④ Autoloader → Bronze (cold path) — the same Kafka topics are consumed into raw immutable Delta partitioned by ingest date. Why: a replayable source of truth for all batch/ML, cheap object storage, exactly-once file tracking.
⑤ Spark (Silver/Gold) — heavy distributed transforms build cleaned, conformed, aggregated tables and features. Why Spark: petabyte-scale batch, ACID via Delta, and it shares the lakehouse with ML.
⑥ Online store (DynamoDB/Redis) — sub-10ms key lookups for the personalization/search/cart APIs. Why: user-facing systems can't query the lake; batch/stream precompute features here.
⑦ Warehouse + OLAP + Search — Redshift/Snowflake for BI, Druid for real-time slice-and-dice dashboards, Elasticsearch for the search index. Why three: different read patterns (SQL analytics vs sub-second aggregation vs text/relevance) need different engines.

Transformation chain B — orders (CDC change → conformed fact)

1
Decode CDC
Avro change events (before/after images, op type, LSN) → rows.
CDC msg → {op:I/U/D, order fields, commit_lsn}
2
Order & dedup
Sort changes per order by commit LSN; drop duplicate deliveries.
change events → ordered, distinct changes per order_id
3
Apply (MERGE)
Upsert current state; handle deletes as tombstones — idempotent.
ordered changes → silver.order_current (latest state)
4
Enrich & conform
Join to customer, product, store dims (SCD2 as of order time); resolve surrogate keys.
order_current → conformed with dim keys valid at event time
5
Derive facts
Compute measures at grain "one row per order line": qty, gross, discount, tax, margin.
conformed orders → gold.fct_order_line (additive measures)
6
Aggregate & reconcile
Roll up to daily sales, cohorts, category; reconcile totals to the payment ledger.
fct_order_line → sales marts + reconciliation status
Why MERGE + LSN ordering matters: an order moves created→paid→shipped→delivered→returned; CDC emits each transition, possibly out of order and more than once. Ordering by LSN + idempotent MERGE guarantees the "current" table is always correct and any replay reproduces it exactly.

Customer 360 — stitching everything into one profile

The golden customer profile joins behavior (clickstream), transactions (orders/payments), service (returns/cases), and marketing consent. Identity resolution runs deterministic matching first (login/email/phone/loyalty id), then probabilistic (device + address + payment fingerprint), producing a master_customer_id with an SCD2 history and a reversible crosswalk (source ids → master). Gold serves dim_customer, cust_360 (RFM, LTV, segments, next-best-action features) to marketing, personalization, and service — with GDPR erasure propagated through the crosswalk.

Storage layout & serving

Lakehouse layout

Bronze partitioned by ingest_dt (immutable, replay). Silver partitioned by business date + Z-ordered/liquid-clustered on entity keys (user, item, order). Gold dimensional marts + feature tables, compacted (OPTIMIZE) and pruned; hot 90 days on standard storage, older tiered to cheap archive.

Serving engines (each for a reason)

DynamoDB/Redis — online features + session state for <10ms API reads. Elasticsearch — product search index (text + relevance). Druid — real-time OLAP for ops dashboards (sub-second group-bys over recent data). Redshift/Snowflake — SQL warehouse for BI/finance and cohort analysis. Feature Store — training/serving parity for rec/search/fraud models. Reverse-ETL — segments pushed back to CRM/ad platforms.

Failure & scale notes (what actually breaks here)

Hot-key skew (a viral product / mega-seller) hard

One item/seller key overwhelms a partition/task. Salt the hot key, broadcast small dims, enable AQE skew handling, and for Kafka pre-plan partition counts. The tell: a straggler task or a lagging Kafka partition.

Prime-Day 10× surge hard

Kafka absorbs the burst (buffer); Flink/Spark autoscale; online store pre-warmed; batch backfills catch up after. The log decoupling is exactly why a surge doesn't topple the platform.

Online/offline feature skew hard

If training features differ from serving features, models silently degrade. A shared Feature Store with one definition computed for both offline (Delta) and online (DynamoDB) guarantees parity.

Late/duplicate clicks distort trending medium

Watermarks bound lateness; dedup on event_id; very-late events correct via batch reprocessing. Trending counters use event-time, not processing-time.

Stage-by-stage

① Gateway — authenticates devices, validates pings, and publishes to Kafka partitioned by geo-zone (so a city's data is colocated and ordered). Why geo-partition: marketplace decisions are per-zone, so colocating a zone's pings makes stateful stream ops local and fast.
② Kafka — durable geo-partitioned log; buffers mobile bursts, decouples the marketplace loop from analytics. Schema Registry governs the ping/trip schemas.
③ Flink (hot marketplace loop) — geo-indexes each ping to an H3 hexagon, map-matches GPS to roads, maintains per-zone supply/demand state, computes surge and ETA features, and drives live matching — all sub-second, event-time, watermarked. Why Flink: low-latency stateful geospatial windows are its sweet spot.
④ Online store (Redis/geo KV) — current zone supply/demand, surge multipliers, driver positions for <10ms reads by the matching/pricing APIs.
⑤ Autoloader → Bronze → Spark — the same streams land raw in Delta; batch assembles complete trips, cleans geo, and builds petabyte history + ML features (ETA, pricing, fraud, driver quality).
⑥ Druid + Warehouse — Druid powers city-ops real-time dashboards (sub-second geo/time slice); the warehouse serves finance, driver earnings, and cohort analytics.

Failure & scale notes

Late/out-of-order GPS from bad networks hard

Process on event-time with watermarks sized to real device lateness; on-device buffering preserves original timestamps; very-late pings correct trip records in batch. Never use processing-time for geo windows.

Hot zone (concert/airport surge) hard

One zone's partition spikes. H3 sub-cell partitioning spreads load; Flink state is keyed by cell; autoscale consumers. The geo-partition design localizes the blast radius to that city.

State store growth in streaming medium

Per-zone/per-trip state must be bounded by watermarks and TTLs, checkpointed to durable storage for exactly-once recovery. Monitor state size; unbounded state is the classic streaming OOM.

Stage-by-stage

① Kafka (exactly-once) — idempotent producers + transactions so a transaction event is never duplicated or lost, keyed by card/account for per-account ordering. Why: the whole platform's correctness starts with an exactly-once, ordered log.
② Flink (fraud hot path) — enriches with account/device features (from the online store), computes velocity/behavioral features, and scores rules + ML in <300ms, returning allow/deny/step-up. Why Flink: low-latency stateful scoring; state checkpointed for exactly-once.
③ CDC + Bronze/Silver — ledger CDC and the txn stream land in Delta; Silver dedups and MERGEs to the current transaction state, then reconciles to the ledger. Why: idempotent, replayable financial truth with ACID.
④ Ledger reconciliation — continuous control-total and row-level reconciliation of stream vs ledger vs settlement files; any break halts and alerts. Why: this is the correctness guarantee regulators and finance demand.
⑤ Immutable audit store (WORM) — every transaction + decision written to write-once storage with time travel for 7–10 years. Why: non-repudiable audit for compliance and disputes.

Failure & scale notes

A reconciliation break (stream ≠ ledger) hard

Halt, don't publish. Drill control totals + row hashes across stream/ledger/settlement to localize where money diverged, fix root cause, reprocess idempotently, re-reconcile. This is the core discipline of a payments platform.

Exactly-once under retries/failures hard

Cross-Cutting Patterns & Tech Reference

The reusable moves behind all five blueprints — recognize these and any new mega-system becomes assembly, not invention.

The patterns that repeat in every giant

1. Durable log as the backbone. Every platform puts Kafka/Kinesis/PubSub between producers and everything else — decouple, buffer, replay. The raw log is sacred: it's how you reprocess history for new metrics/models.

2. Hot/cold split (Lambda) or replayable single path (Kappa). Real-time serving needs precomputed features in an online store; heavy history/ML lives in the lakehouse. They meet at the online store, which batch backfills and stream keeps fresh.

3. Decision-as-data. Every real-time decision (price, surge, fraud score, bid, recommendation) is logged back to the log → lakehouse — for audit, replay, and model training. The system learns from its own outputs.

4. Precompute for serving. Nobody scans a petabyte in an API call. Features → online KV; dashboards → real-time OLAP (Druid/Pinot) pre-rollups; search → inverted index. Match the engine to the read pattern.

5. Idempotency + exactly-once + reconciliation. At-least-once delivery + dedup on a stable id + MERGE/upsert = correct, replayable state. Reconciliation (counts + control totals) proves nothing was lost or invented — mandatory for money and billing.

6. Event-time + watermarks. Real data arrives late and out of order. Correct systems window on event-time with watermarks (bounded state) and reprocess late data in batch — never trust processing-time for correctness.

7. Feature store parity. One feature definition, computed for both offline (training) and online (serving), prevents the silent model decay of train/serve skew.

8. Medallion refinement. Bronze (raw, replayable) → Silver (clean, deduped, conformed) → Gold (aggregates/features/marts). Contain failures per layer; each is independently reprocessable.
