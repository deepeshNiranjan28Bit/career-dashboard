SQL — The #1 Filter

If you can write clean window functions and reason about join cardinality under pressure, you pass most first rounds. This is the highest-ROI topic in the whole doc.

Practical theory you must be able to say out loud

Logical query execution order

SQL is written SELECT … FROM … WHERE … GROUP BY … HAVING … ORDER BY but runs in a different order. Knowing this explains 90% of "why doesn't this work" bugs:

FROM/JOIN → WHERE → GROUP BY → HAVING → SELECT (aliases created here) → DISTINCT → ORDER BY → LIMIT.

Consequences interviewers love: you can't use a SELECT alias in WHERE (alias doesn't exist yet) but you can in ORDER BY. WHERE filters rows before grouping; HAVING filters groups after. Window functions run after GROUP BY/HAVING but before the final SELECT-level DISTINCT/ORDER BY — which is why you often need a subquery/CTE to filter on a window result.

Joins & cardinality — the thing that separates real engineers

A join is a Cartesian product followed by a filter. It fans out whenever the join key is not unique on the "many" side. Duplicated keys silently inflate SUM/COUNT — the #1 source of wrong numbers in pipelines. Before trusting an aggregate, know each join's cardinality: 1:1, 1:many, or many:many. INNER keeps only matches; LEFT keeps all left rows (unmatched right = NULL); FULL OUTER keeps both sides (MySQL lacks it — emulate with LEFT UNION RIGHT).

NULL & three-valued logic

NULL is "unknown," not a value. Any comparison with NULL is UNKNOWN, and a row is kept only when the predicate is TRUE — so WHERE bonus <> 100 silently drops NULL bonuses. Use IS NULL, COALESCE, and beware NOT IN (subquery) that returns any NULL — it yields zero rows. Aggregates skip NULLs: COUNT(col) ignores NULLs, COUNT(*) doesn't.

Window functions — aggregate without collapsing

A window function computes across a set of rows related to the current row without collapsing them. Syntax: func() OVER (PARTITION BY … ORDER BY … ROWS/RANGE …). Know cold: ROW_NUMBER (unique 1,2,3), RANK (gaps on ties), DENSE_RANK (no gaps), LAG/LEAD (previous/next row), and running totals with a frame. This is the single most-tested SQL construct for DE.

-- top 3 earners per department
SELECT * FROM (
  SELECT emp, dept, salary,
         DENSE_RANK() OVER (PARTITION BY dept ORDER BY salary DESC) AS rnk
  FROM employees
) t
WHERE rnk <= 3;

The frame clause (the detail people miss)

With ORDER BY in a window and no explicit frame, the default is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW — which, on ties, includes all peer rows, not just up to the current one. For a true running total row-by-row, be explicit: ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW.

Say-it-cold checklist: logical execution order · join fan-out · NULL/3VL · the four ranking functions · LAG/LEAD · running total with explicit ROWS frame · WHERE vs HAVING · UNION vs UNION ALL · correlated vs non-correlated subquery · gaps-and-islands pattern.
Live-coding trap: filtering on a window function directly (WHERE ROW_NUMBER() OVER(...) = 1) is illegal — windows are computed after WHERE. Wrap in a subquery/CTE and filter on the alias.

Patterns worth memorizing

Deduplicate keep-latest: ROW_NUMBER() OVER(PARTITION BY id ORDER BY updated_at DESC)=1. Nth highest: DENSE_RANK()…=N. Running total / moving avg: window with explicit ROWS frame. Gaps-and-islands (consecutive streaks): subtract ROW_NUMBER() from a date/sequence; equal difference = same island. Pivot: SUM(CASE WHEN k='a' THEN v END). Month-over-month: LAG(metric) OVER(ORDER BY month) then compute delta.

Interview Q&A (25)

Q1. Explain the logical order of execution of a SELECT. basic

FROM/JOIN → WHERE → GROUP BY → HAVING → SELECT (aliases materialize here) → DISTINCT → ORDER BY → LIMIT. This is why a SELECT alias can't be used in WHERE but can in ORDER BY, and why WHERE filters rows while HAVING filters groups.

Q2. WHERE vs HAVING? basic

WHERE filters individual rows before grouping; HAVING filters groups after aggregation. You can't use an aggregate in WHERE; you can in HAVING. Push filters into WHERE so fewer rows reach GROUP BY.

Q3. UNION vs UNION ALL? basic

UNION removes duplicates (implicit DISTINCT → a sort/hash cost); UNION ALL keeps everything and is much faster. Default to UNION ALL unless you truly need dedup. Both need matching column count and compatible types, aligned by position.

Q4. If A and B each have 100 rows, what row counts can INNER/LEFT/CROSS produce? medium

INNER: 0–10,000 depending on key cardinality (1:1 gives ≤100, 1:many multiplies). LEFT: ≥100 (every A row survives), more if A matches multiple B rows. CROSS: exactly 10,000. The point: joins fan out when the join key isn't unique on the many side.

Q5. ROW_NUMBER vs RANK vs DENSE_RANK. basic

All assign order within a partition. ROW_NUMBER: always unique (1,2,3,4). RANK: ties share a rank and leave gaps (1,1,3). DENSE_RANK: ties share a rank, no gaps (1,1,2). Use ROW_NUMBER for dedup, DENSE_RANK for "top-N including ties."

Q6. Find the 2nd highest salary — multiple ways. medium

Robust: SELECT MAX(salary) FROM emp WHERE salary < (SELECT MAX(salary) FROM emp). Or window: DENSE_RANK() OVER(ORDER BY salary DESC)=2. Or OFFSET 1 LIMIT 1 on distinct salaries. Mention DENSE_RANK handles ties correctly and returns NULL-safe empty if fewer than 2 distinct values.

Q7. Deduplicate a table keeping the latest record per key. medium 
WITH r AS (
  SELECT *, ROW_NUMBER() OVER (
    PARTITION BY id ORDER BY updated_at DESC) rn
  FROM t)
SELECT * FROM r WHERE rn = 1;

ROW_NUMBER (not RANK) so ties still collapse to one row. Choose a deterministic tiebreaker in ORDER BY.

Q8. Running total of daily sales. medium 
SELECT day, amt,
  SUM(amt) OVER (ORDER BY day
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS running
FROM sales;

Give the explicit ROWS frame — the default RANGE frame includes all tied peers, which is wrong for true row-by-row running totals.

Q9. Month-over-month growth %. medium 
SELECT month, rev,
  LAG(rev) OVER (ORDER BY month) AS prev,
  ROUND(100.0*(rev - LAG(rev) OVER (ORDER BY month))
        / LAG(rev) OVER (ORDER BY month), 2) AS mom_pct
FROM monthly;

LAG pulls the previous row. Guard divide-by-zero/NULL for the first month with NULLIF/COALESCE.

Q10. Gaps-and-islands: find consecutive-day login streaks. hard

Trick: for each user, date - ROW_NUMBER() OVER(PARTITION BY user ORDER BY date) is constant within a consecutive run. Group by that constant to get each streak's start, end, and length.

SELECT user, MIN(d) start, MAX(d) end, COUNT(*) len FROM (
  SELECT user, d,
    d - INTERVAL ROW_NUMBER() OVER(PARTITION BY user ORDER BY d) DAY AS grp
  FROM logins) x
GROUP BY user, grp;
Q11. Why does WHERE col <> 5 drop NULL rows, and how to include them? medium

NULL <> 5 evaluates to UNKNOWN, and only TRUE rows are kept, so NULLs vanish. Include them explicitly: WHERE col <> 5 OR col IS NULL. Same reason NOT IN with a NULL in the list returns nothing.

Q12. Correlated vs non-correlated subquery. medium

Non-correlated runs once, independent of the outer query. Correlated references the outer row and conceptually runs per outer row (optimizers often rewrite it to a join/semi-join). EXISTS is a correlated semi-join and usually beats IN for large/NULL-prone sets.

Q13. EXISTS vs IN vs JOIN for "customers who ordered." medium

All can express it. EXISTS short-circuits on first match and is NULL-safe. IN is fine but breaks with NULLs in the subquery for NOT IN. A JOIN can duplicate customers if a customer has many orders (use DISTINCT or a semi-join). Prefer EXISTS/semi-join for existence checks.

Q14. What is a CTE and when is it better than a subquery? basic

A CTE (WITH … AS) names a query block for readability and reuse within one statement. Better for multi-step logic and self-documentation; recursive CTEs handle hierarchies/graphs. Note: in some engines a CTE is an optimization fence (not inlined), so for hot paths test vs an inline subquery.

Q15. Write a recursive CTE to walk an employee→manager hierarchy. hard 
WITH RECURSIVE org AS (
  SELECT id, name, manager_id, 1 lvl
  FROM emp WHERE manager_id IS NULL
  UNION ALL
  SELECT e.id, e.name, e.manager_id, o.lvl+1
  FROM emp e JOIN org o ON e.manager_id = o.id)
SELECT * FROM org;

Anchor member seeds the roots; recursive member joins back to the CTE until no new rows. Always ensure a termination condition to avoid infinite loops.

Q16. Pivot rows to columns without PIVOT syntax. medium 
SELECT dept,
  SUM(CASE WHEN quarter='Q1' THEN amt END) q1,
  SUM(CASE WHEN quarter='Q2' THEN amt END) q2
FROM sales GROUP BY dept;

Conditional aggregation is the portable pivot. Reverse (unpivot) with UNION ALL of one column per source column.

Q17. What's an index and when does it hurt? medium

An index (usually a B+tree) is a sorted structure that turns a full scan into a seek — great for selective lookups, range scans, and ORDER BY/JOIN keys. It hurts on writes (every insert/update maintains it), on low-selectivity columns (optimizer may ignore it), and when you wrap the column in a function (WHERE DATE(ts)=… can't use an index on ts). Rewrite as a sargable range instead.

Q18. What does "sargable" mean and why care? medium

Search-ARGument-able: a predicate the engine can satisfy via an index seek. WHERE ts >= '2026-01-01' AND ts < '2026-02-01' is sargable; WHERE YEAR(ts)=2026 or WHERE col+0=5 or leading-wildcard LIKE '%x' are not — they force scans. Keep the indexed column bare on one side.

Q19. How do you read/act on an EXPLAIN plan? hard

Look for: access type (full table scan vs index seek/range), estimated vs actual rows (bad estimates → stale stats), join algorithm and order, and expensive sorts/temp tables. Fixes: add/adjust indexes, make predicates sargable, update statistics, reduce rows earlier, or restructure the join. Big gap between estimated and actual rows is the classic tell of a bad plan.

Q20. INNER JOIN with a filter in ON vs WHERE for a LEFT JOIN — difference? hard

For INNER JOIN it's equivalent. For LEFT JOIN it's not: a condition in ON filters the right table before the join (unmatched left rows survive with NULLs), while the same condition in WHERE runs after and eliminates the NULL-extended rows — effectively turning it into an INNER JOIN. Classic trap.

Q21. De-duplicate detection: how to find duplicate rows? basic 
SELECT key_cols, COUNT(*) c
FROM t GROUP BY key_cols HAVING COUNT(*) > 1;

To delete keeping one, use ROW_NUMBER per key and delete rn>1. Always confirm the "true key" first — duplicates often reveal a broken upstream join.

Q22. COUNT(*) vs COUNT(col) vs COUNT(DISTINCT col). basic

COUNT(*) counts all rows including NULLs. COUNT(col) counts non-NULL values only. COUNT(DISTINCT col) counts distinct non-NULL values (more expensive — needs a sort/hash). This trips people up in data-quality checks.

Q23. How would you find records in table A not in table B? medium

Three ways: LEFT JOIN … WHERE b.key IS NULL (anti-join, usually best), NOT EXISTS (NULL-safe), or EXCEPT. Avoid NOT IN if B's key can be NULL — it returns zero rows.

Q24. What is a window frame and when does the default bite you? hard

The frame defines which rows around the current row the function sees. With ORDER BY and no explicit frame, default is RANGE UNBOUNDED PRECEDING TO CURRENT ROW, which includes all rows tied on the ORDER BY value — so a running SUM jumps at ties. Use ROWS for exact row-by-row behavior.

Q25. You have a slow analytical query on a billion-row table. Walk me through tuning it. hard

(1) Read the plan — find the scan/skew/sort. (2) Reduce data early: partition pruning (filter on partition column), column pruning (avoid SELECT *), predicate pushdown. (3) Make predicates sargable and ensure partitioning/clustering matches the filter. (4) Pre-aggregate or materialize if reused. (5) On MPP/warehouse: check distribution/clustering keys to avoid shuffle-heavy joins; broadcast small dimensions. (6) Validate with before/after runtimes and bytes scanned, not guesses.

Python for Data Engineering

DE Python interviews aren't LeetCode-hard. They test whether you understand how Python actually behaves — mutability, iteration, the GIL — and whether you write clean, testable data code.

Practical theory you must be able to say out loud

Everything is an object; names are references

Variables are names bound to objects, not boxes holding values. Assignment binds a name to an object; passing to a function passes the reference. Mutating a mutable object (list, dict) inside a function affects the caller; rebinding the name doesn't. This "pass by object reference" model explains most Python surprises.

Mutable vs immutable

Immutable: int, float, str, tuple, frozenset, bytes. Mutable: list, dict, set, bytearray. Only immutable (hashable) objects can be dict keys / set members. The classic trap: a mutable default argument is created once and shared across calls.

def add(x, acc=[]):   # BUG: acc persists across calls
    acc.append(x); return acc
def add(x, acc=None): # FIX
    acc = acc or []; acc.append(x); return acc

The iteration protocol — iterables vs iterators vs generators

An iterable has __iter__ (list, dict, file). An iterator has __next__ and is consumed once. A generator (function with yield, or a genexpr) is a lazy iterator that produces values on demand — essential in DE for streaming large files without loading them into memory.

def read_big(path):
    with open(path) as f:
        for line in f:          # lazy, one line at a time
            yield line.strip()

The GIL and true concurrency

CPython's Global Interpreter Lock lets only one thread execute Python bytecode at a time. So threads help I/O-bound work (network, disk, DB calls — the GIL is released during I/O) but not CPU-bound work. For CPU parallelism use multiprocessing (separate interpreters) or push compute into libraries that release the GIL / run natively (NumPy, PySpark). In DE, most heavy lifting is delegated to Spark/SQL engines, so the GIL rarely bites directly.

Data structures & complexity

dict/set give O(1) average lookup (hash tables) — use a set for membership tests, not a list (O(n)). list append is amortized O(1), insert-at-front is O(n) (use collections.deque). Comprehensions are faster and clearer than manual loops. Know collections.Counter, defaultdict, and itertools (groupby, chain, islice) — they show fluency.

Say-it-cold checklist: pass-by-object-reference · mutable-default trap · list vs generator (memory) · dict/set O(1) · GIL & when threads help · context managers (with) · exception handling · *args/**kwargs · why pandas breaks at scale & when to reach for Spark.
Common trap: a = b = [] makes both names point to the same list. And list_of_lists = [[]]*3 creates three references to one inner list — mutating one mutates all.

Production-Python habits that read as senior

Type hints + docstrings; small pure functions; config via env vars not hardcoding; logging not print; explicit exceptions with context; idempotent writes; if __name__ == "__main__" entrypoints; unit tests with pytest; virtual environments (venv/poetry) and pinned dependencies. Mentioning these unprompted signals you write real pipeline code, not scripts.

Interview Q&A (25)

Q1. Is Python pass-by-value or pass-by-reference? medium

Neither exactly — it's "pass by object reference." The function receives a reference to the same object. Mutating a mutable arg is visible to the caller; rebinding the parameter name is not. Demonstrate with a list append (visible) vs x = x + 1 (not).

Q2. List vs tuple — when use which? basic

List is mutable, variable-length, for homogeneous sequences you'll change. Tuple is immutable, hashable (can be a dict key/set member), slightly lighter, and signals "fixed record." Use tuples for fixed structures and dict keys; lists for collections you mutate.

Q3. What's the mutable-default-argument trap? medium

Default values are evaluated once at function definition, so a def f(x, acc=[]) shares one list across all calls, accumulating state. Fix with acc=None then acc = acc or [] inside.

Q4. Explain generators and why they matter in DE. medium

A generator yields values lazily, holding only one at a time in memory, so you can process a 100GB file or an infinite stream without loading it. It also enables pipelines (chain generators). Contrast with a list comprehension that materializes everything.

Q5. What is the GIL and how do you work around it? hard

The Global Interpreter Lock serializes Python bytecode execution to one thread at a time. Threads still help I/O-bound tasks (GIL released during I/O). For CPU-bound parallelism use multiprocessing, or delegate to native libraries (NumPy/PySpark) that release the GIL. In DE, compute is usually pushed to Spark/SQL, so it's mostly moot.

Q6. threading vs multiprocessing vs asyncio — pick one. medium

I/O-bound, many waits: asyncio (single thread, event loop) or threading. CPU-bound: multiprocessing (bypasses GIL). asyncio is best for thousands of concurrent network calls; threading is simpler for a handful of blocking I/O calls; multiprocessing for parallel computation.

Q7. How do context managers work? medium

with calls __enter__ on entry and guarantees __exit__ on exit (even on exception) — used to release resources: files, DB connections, locks, Spark sessions. Write one with contextlib.contextmanager and a yield. Prevents leaks that plague long-running pipelines.

Q8. What are decorators and where would you use one in a pipeline? medium

A decorator wraps a function to add behavior without changing its body. In DE: retry-on-failure, timing/metrics, logging, caching, auth. @functools.wraps preserves metadata. Airflow's @task is a decorator you already use conceptually.

Q9. *args and **kwargs? basic

*args captures extra positional args as a tuple; **kwargs captures extra keyword args as a dict. Used for flexible wrappers/decorators and forwarding args to another function: inner(*args, **kwargs).

Q10. How would you read a 50GB CSV in Python? medium

Don't load it whole. Options: iterate line-by-line (generator), pandas.read_csv(chunksize=…) to process in chunks, use a columnar/streaming lib (PyArrow, Polars), or — realistically at that scale — process it in Spark. Emphasize bounded memory and, ideally, converting to Parquet.

Q11. When does pandas break, and what then? medium

pandas is single-machine, in-memory — it breaks when data exceeds RAM (~a few GB comfortably) or needs distributed compute. Then move to Spark/PySpark (distributed), DuckDB/Polars (fast single-node, out-of-core), or push aggregation into the warehouse. pandas stays great for small data, prototyping, and last-mile transforms.

Q12. dict vs list for membership testing? basic

x in some_set/dict is O(1) average (hash lookup); x in some_list is O(n) (linear scan). For repeated membership checks, build a set first. This is a frequent source of accidental O(n²) code.

Q13. Shallow vs deep copy. medium

Shallow copy (list(x), copy.copy) duplicates the outer container but shares nested objects. Deep copy (copy.deepcopy) recursively duplicates everything. Matters when you mutate nested structures and don't want to affect the original.

Q14. How do you handle exceptions cleanly in a pipeline task? medium

Catch specific exceptions (not bare except:), add context, log with stack trace, and decide: retry (transient), skip/quarantine the record (bad data), or fail fast (systemic). Use try/except/else/finally; finally for cleanup. Re-raise or wrap in a custom exception so upstream orchestration sees the failure.

Q15. What are Python's iterables vs iterators? medium

An iterable can produce an iterator (__iter__); an iterator produces the next value (__next__) and raises StopIteration when done. A list is iterable (re-iterable); the iterator from it is single-use. Generators are iterators.

Q16. Explain list/dict/set comprehensions and a genexpr. basic

[f(x) for x in xs if c(x)] builds a list eagerly; {k:v …} a dict; {x …} a set; (f(x) for x …) a lazy generator. Comprehensions are faster and clearer than append loops; use a genexpr when you only iterate once.

Q17. What does @staticmethod / @classmethod do? medium

@classmethod receives the class (cls) — used for alternative constructors (from_config). @staticmethod receives neither self nor cls — just a namespaced utility. Regular methods receive the instance.

Q18. How would you make a function retry on transient failures? medium

A retry decorator with exponential backoff and a cap, retrying only on transient exceptions (timeouts, 5xx), not on logic errors. Use the tenacity library in production. Combine with idempotency so retries are safe.

Q19. What is __name__ == "__main__" for? basic

It guards code that should run only when the file is executed directly, not when imported as a module. Essential so importing your pipeline module for its functions doesn't trigger a full run — and required for multiprocessing on some platforms.

Q20. How do you test data pipeline code? medium

Structure logic into small pure functions, then unit-test transforms with pytest on tiny fixture inputs (including edge cases: nulls, empty, dupes). Add data-quality assertions on outputs, and integration tests on a sample. Mock external systems. For Spark, test transform functions on small local DataFrames.

Q21. Explain memory: why is a generator lighter than a list of the same data? medium

A list stores every element in memory simultaneously; a generator computes and hands off one element at a time, keeping only current state. So a list of 10M ints costs hundreds of MB; a generator costs a constant few bytes — at the cost of being single-pass and not indexable.

Q22. What are Python's number/precision pitfalls in data work? medium

Floats are binary IEEE-754 — 0.1+0.2 != 0.3. For money/exact decimals use decimal.Decimal. Beware integer vs float division, and silent precision loss when casting. In pipelines, define numeric types explicitly (schemas) rather than inferring.

Q23. How do you connect Python to a database safely? medium

Use a driver/ORM with a connection pool and context managers, parameterized queries (never string-format SQL — SQL injection + type bugs), fetch in batches for large results, and secrets from a vault/env not code. Close connections in finally/with.

Q24. What is duck typing and how does it affect design? basic

"If it walks like a duck…" — Python cares about behavior (methods) not declared type. Lets you write generic code against protocols (anything with .read()). Downside: errors surface at runtime, so type hints + tests compensate.

Q25. Walk me through structuring a small ETL script well. hard

Separate concerns: extract (I/O in, returns raw), transform (pure functions, unit-tested), load (idempotent write). Config from env; logging throughout; explicit schema; validate inputs and outputs (row counts, nulls); wrap in try/except with clear failure signals; make the whole run idempotent (safe to re-run for the same date/partition); expose a single entrypoint under __main__. That structure is what interviewers want to hear.

Data Modeling & Warehousing

Asked in almost every DE interview: star schema, fact vs dimension, SCD Type 1 vs 2, normalization vs denormalization. Have crisp definitions and be ready to model a domain live.

Practical theory you must be able to say out loud

OLTP vs OLAP — why we model differently

OLTP (transactional apps) optimizes for many small reads/writes → highly normalized, row-store, integrity-first. OLAP (analytics/warehouse) optimizes for large scans and aggregations → denormalized, columnar, read-first. The warehouse exists because running analytics on the OLTP database would both be slow and hammer production.

Normalization vs denormalization

Normalization (1NF→3NF) removes redundancy so each fact is stored once — great for OLTP integrity, but analytics queries need many joins. Denormalization deliberately duplicates data to cut joins and speed reads — the warehouse trade. 1NF: atomic values. 2NF: no partial dependency on part of a composite key. 3NF: no transitive dependency (non-key depending on non-key).

Dimensional modeling (Kimball) — star schema

A fact table holds measurements (numeric, additive) at a defined grain, with foreign keys to dimensions. A dimension table holds descriptive context (who/what/where/when). A star schema = one fact surrounded by denormalized dimensions. A snowflake normalizes dimensions into sub-tables (more joins, less duplication) — usually avoid unless dimensions are huge.

Grain is the first decision: "one row per order line per day." Get grain wrong and every metric is wrong. Fact types: transaction (one row per event), periodic snapshot (one row per entity per period), accumulating snapshot (one row per process, updated as it progresses).

Slowly Changing Dimensions (SCD)

How you handle dimension attributes that change over time. Type 1: overwrite — no history (fix a typo). Type 2: add a new row with effective/expiry dates + a current flag (or surrogate key) — full history, the interview favorite. Type 3: add a "previous value" column — limited history. Type 2 is why you use surrogate keys (a warehouse-generated integer) instead of the natural business key.

Additivity of facts

Additive measures sum across all dimensions (revenue). Semi-additive sum across some but not time (account balance — sum across accounts, not across days; use last value). Non-additive can't be summed (ratios, percentages — recompute from components).

Say-it-cold checklist: OLTP vs OLAP · grain · fact vs dimension · star vs snowflake · SCD1 vs SCD2 (and why surrogate keys) · additive/semi/non-additive · normalization forms · fact-table types · conformed dimensions · warehouse vs lake vs lakehouse.
Trap: "What's the grain of this fact table?" If you can't answer in one sentence, your model is wrong. Also: never put a measure that changes meaning at a different grain (e.g., unit price) as additive.

Warehouse vs Lake vs Lakehouse (one paragraph)

Warehouse: structured, schema-on-write, SQL, governed, great BI performance (Snowflake, Redshift, BigQuery). Lake: cheap object storage holding any format, schema-on-read, flexible but easily a "swamp" without governance (ADLS/S3 + Parquet). Lakehouse: lake storage + a table format (Delta/Iceberg/Hudi) that adds ACID transactions, schema enforcement, time travel, and upserts — warehouse reliability on lake economics (Databricks). The medallion architecture (bronze raw → silver cleaned/conformed → gold business-ready) is how you organize a lakehouse.

Interview Q&A (25)

Q1. OLTP vs OLAP? basic

OLTP = transactional, many small concurrent reads/writes, normalized, row-oriented, integrity-focused (the app DB). OLAP = analytical, large scans/aggregations, denormalized, columnar, read-optimized (the warehouse). Different workloads → different modeling and storage.

Q2. What is a star schema? basic

A central fact table (measures + foreign keys) surrounded by denormalized dimension tables. Simple, few joins, fast for BI. Snowflake schema normalizes dimensions further into sub-tables — more joins, less redundancy.

Q3. Fact vs dimension table? basic

Fact = numeric measurements at a grain (sales amount, quantity) with FKs to dimensions; typically tall and thin, append-heavy. Dimension = descriptive context (customer, product, date); shorter and wide, changes slowly. Facts answer "how much," dimensions answer "who/what/where/when."

Q4. Explain SCD Type 1 vs Type 2. medium

Type 1 overwrites the attribute — no history, used for corrections. Type 2 inserts a new dimension row on change with effective/end dates and a current flag (and a new surrogate key), preserving full history so facts join to the value that was true at event time. Type 3 keeps a single "previous value" column.

Q5. Why use surrogate keys instead of natural keys? medium

To support SCD Type 2 (the same business key has multiple versioned rows), to insulate the warehouse from source-key changes/reuse, to enable fast integer joins, and to handle multi-source or missing keys. Keep the natural/business key as an attribute for lineage.

Q6. What is the "grain" of a fact table and why does it matter first? medium

Grain is what one row represents ("one row per order line item"). It's the first design decision because it determines which dimensions attach and whether measures are additive. Mixed or wrong grain double-counts or loses detail. State it as a single sentence before modeling anything.

Q7. Additive, semi-additive, non-additive measures? medium

Additive: sum across all dimensions (revenue, quantity). Semi-additive: sum across some but not time (balances, inventory — take last/average over time). Non-additive: can't sum meaningfully (ratios, percentages, unit price) — store components and recompute.

Q8. Normalize vs denormalize — when each? medium

Normalize for OLTP: eliminate redundancy, protect integrity, cheap writes. Denormalize for OLAP: duplicate to remove joins and speed reads. Warehouses lean denormalized (star schema); source systems lean normalized. It's a read-speed vs write-integrity/storage trade.

Q9. Explain 1NF, 2NF, 3NF quickly. medium

1NF: atomic column values, no repeating groups. 2NF: 1NF + no non-key column depends on only part of a composite key. 3NF: 2NF + no transitive dependency (non-key depending on another non-key). Mnemonic: each non-key depends on "the key, the whole key, and nothing but the key."

Q10. Warehouse vs data lake vs lakehouse? medium

Warehouse: structured, schema-on-write, governed SQL, fast BI. Lake: any format on cheap object storage, schema-on-read, flexible but ungoverned. Lakehouse: lake storage + a transactional table format (Delta/Iceberg) giving ACID, schema enforcement, upserts, time travel — warehouse features on lake cost.

Q11. What is the medallion (bronze/silver/gold) architecture? basic

Bronze = raw ingested data as-is (immutable landing, replayable). Silver = cleaned, deduplicated, conformed, joined. Gold = business-level aggregates/marts serving BI and ML. Each layer adds quality and is independently reprocessable.

Q12. Conformed dimensions — what and why? medium

A dimension shared consistently across multiple fact tables/marts (same keys and meaning) — e.g., one Date or Customer dimension used everywhere. They let you compare metrics across processes ("drill across") and are the backbone of an enterprise bus-matrix model.

Q13. How do you handle a late-arriving dimension? hard

Insert a placeholder ("inferred member") dimension row with the natural key and unknown attributes so the fact can load with a valid FK, then update it (Type 1) or version it (Type 2) when the real record arrives. Prevents facts from being dropped or orphaned.

Q14. Star vs snowflake — trade-off? medium

Star denormalizes dimensions: fewer joins, faster and simpler queries, some redundancy. Snowflake normalizes them into sub-tables: less storage/redundancy, more joins, more complexity. Default to star; snowflake only when a dimension is genuinely large or shared hierarchies demand it.

Q15. Transaction vs periodic-snapshot vs accumulating-snapshot fact. hard

Transaction: one row per event (a sale). Periodic snapshot: one row per entity per period (daily account balance). Accumulating snapshot: one row per process instance updated as it moves through milestones (order → shipped → delivered, with multiple date FKs). Pick by the business question.

Q16. What is a factless fact table? medium

A fact table with no measures — just the FKs recording that an event/relationship occurred (student attendance, promotion coverage). You "measure" by counting rows. Useful for coverage and event-tracking questions.

Q17. Kimball vs Inmon? medium

Inmon: top-down, build a normalized (3NF) enterprise warehouse first, then derive dimensional marts. Kimball: bottom-up, build dimensional marts (star schemas) tied together by conformed dimensions. Kimball is faster to value and more common in practice; many modern stacks blend both.

Q18. How would you model a slowly changing customer address with full history? hard

SCD Type 2 on the customer dimension: surrogate key PK, natural customer_id, address attributes, valid_from, valid_to, is_current. On change, expire the current row (set valid_to, is_current=false) and insert a new current row. Facts store the surrogate key valid at event time.

Q19. What is a data mart? basic

A subject-focused subset of the warehouse serving one team/domain (finance, marketing). Smaller, faster, easier to govern. Built from conformed dimensions so marts stay consistent with the enterprise model.

Q20. Degenerate dimension — what is it? medium

A dimension attribute stored in the fact table with no separate dimension table — typically a transaction identifier like order number or invoice number. It's a dimension key with no other attributes worth a table, kept for grouping/tracing.

Q21. How do you decide what becomes a fact vs a dimension attribute? medium

If it's a continuously-valued measurement you'll aggregate (amount, count, duration) → fact. If it's descriptive context you'll filter/group by (category, region, status) → dimension. Numeric values you group by rather than sum (e.g., a rating band) often belong in a dimension.

Q22. What is denormalization's downside in a warehouse? medium

Redundant storage and update anomalies — a changed attribute must be rewritten across many rows. In a warehouse you mitigate by rebuilding/merging dimensions in controlled ETL rather than ad-hoc updates, accepting storage cost for read speed.

Q23. What is a bridge table? hard

A table resolving a many-to-many relationship between a fact and a dimension (or between dimensions) — e.g., an account with multiple customers. It holds the key pairs plus optional allocation/weight factors to avoid double counting.

Q24. How would you model event/clickstream data for analytics? hard

Transaction-grain fact (one row per event) with FKs to user, session, page, and a fine-grained date/time dimension; measures like duration. Often stored as partitioned Parquet/Delta by event date. Aggregate to sessions/funnels in gold. Watch cardinality and late/duplicate events (dedup by event_id).

Q25. Walk me through designing a sales data mart end to end. hard

Clarify the business questions → set grain ("one row per order line") → identify measures (quantity, amount, discount) → identify dimensions (date, customer, product, store, promotion) → choose SCD strategy per dimension (Type 2 for customer/product) → assign surrogate keys → design the fact with FKs + degenerate order number → define conformed date/customer dims → plan the ETL (bronze→silver→gold, incremental by order date, idempotent) → add DQ checks (row counts, referential integrity, additivity). That structured walk is the answer they want.

Database Internals

Enough engine internals to answer indexing, ACID, isolation, and locking questions with confidence — the follow-ups that come after live SQL.

Practical theory you must be able to say out loud

Indexes are B+trees; the index often is the table

Most indexes are B+trees: sorted, balanced, O(log n) lookups, and efficient range scans (leaves are linked). In InnoDB the table is stored as its primary-key B+tree (a clustered index) — rows live in PK order in the leaves. Secondary indexes store the PK value, so a lookup via a secondary index does a second seek into the clustered index (a "bookmark lookup") unless the index is covering (contains all needed columns).

ACID

Atomicity: all-or-nothing (via undo log/rollback). Consistency: constraints hold before/after. Isolation: concurrent transactions don't corrupt each other. Durability: committed data survives crashes (via the write-ahead/redo log). Databases achieve durability by flushing the log before acknowledging commit, then lazily writing data pages.

Isolation levels & the anomalies they prevent

Read Uncommitted (dirty reads possible) → Read Committed (no dirty reads; non-repeatable reads possible) → Repeatable Read (no non-repeatable reads; phantom reads possible in the standard, prevented by InnoDB via gap locks) → Serializable (full isolation, least concurrency). Most systems run Read Committed (default in Postgres/Oracle) or Repeatable Read (default in MySQL/InnoDB). MVCC lets readers see a consistent snapshot without blocking writers.

Locking & deadlocks

Writers take row locks; conflicting writers wait. A deadlock is a cycle of waits (T1 holds A wants B, T2 holds B wants A) — the engine detects it and kills one transaction; you retry. Reduce deadlocks by accessing rows in a consistent order and keeping transactions short.

Interview Q&A (14)

Q1. How does a B+tree index speed up lookups? medium

It keeps keys sorted in a balanced tree, so a lookup is O(log n) instead of scanning every row, and linked leaves make range scans and ORDER BY cheap. The trade: writes must maintain the tree, so indexes cost insert/update/delete time and storage.

Q2. Clustered vs non-clustered (secondary) index? medium

Clustered index defines the physical row order (in InnoDB the PK — the table IS the index). There's only one. Secondary indexes are separate structures pointing back (via the PK) to rows, so they may require a second lookup unless they "cover" the query.

Q3. What is a covering index? medium

An index that contains every column a query needs, so the engine answers entirely from the index without touching the table (no bookmark lookup). Great for hot read paths; costs extra storage and write overhead.

Q4. Explain ACID. basic

Atomicity (all-or-nothing via rollback), Consistency (constraints preserved), Isolation (concurrent txns don't interfere), Durability (committed data survives crashes via the WAL/redo log). Together they guarantee reliable transactions.

Q5. What are the isolation levels and what does each allow? hard

Read Uncommitted: dirty reads. Read Committed: no dirty reads, but non-repeatable reads. Repeatable Read: no non-repeatable reads, but (standard) phantoms. Serializable: none. Higher isolation = less concurrency. Know the default of your engine (MySQL: Repeatable Read; Postgres: Read Committed).

Q6. What is MVCC? hard

Multi-Version Concurrency Control: each transaction sees a consistent snapshot via row versions, so readers don't block writers and writers don't block readers. Old versions are kept (undo log) until no transaction needs them, then garbage-collected. It's how databases give isolation without heavy read locks.

Q7. Dirty read, non-repeatable read, phantom read? hard

Dirty: reading another txn's uncommitted change. Non-repeatable: re-reading the same row returns a different value because another txn committed an update. Phantom: re-running a range query returns new/removed rows because another txn inserted/deleted. Each is prevented at a higher isolation level.

Q8. What causes a deadlock and how do you handle it? medium

A cycle of lock waits between transactions. The engine detects the cycle and aborts one (the victim); the app should catch the error and retry. Prevent by accessing resources in a consistent order, keeping transactions short, and lowering isolation where safe.

Q9. When should you NOT add an index? medium

On low-selectivity columns (few distinct values — the optimizer scans anyway), on small tables, on write-heavy tables where index maintenance dominates, and redundant indexes that duplicate a prefix of another. Every index is a write tax; add only for real read patterns.

Q10. Why can't an index be used on WHERE YEAR(ts)=2026? medium

Wrapping the column in a function makes the predicate non-sargable — the index is on ts, not on YEAR(ts). Rewrite as a range: ts >= '2026-01-01' AND ts < '2027-01-01', which uses the index. (Or add a functional/computed index.)

Q11. Partitioning vs sharding vs replication? medium

Partitioning splits one table into segments (by range/hash) within one database for manageability and pruning. Sharding spreads data across multiple database servers for horizontal scale. Replication copies data to other nodes for read scaling and high availability. They solve different problems and are often combined.

Q12. What is a composite index and does column order matter? medium

An index on multiple columns. Order matters: it supports queries filtering on a left-prefix (index on (a,b,c) helps a, a,b, a,b,c, not b alone). Put the most selective / most-filtered column first, and match the ORDER BY to avoid a sort.

Q13. OLTP database vs a columnar warehouse — storage difference? medium

OLTP is row-store: a row's columns are contiguous, ideal for reading/writing whole records. Warehouses are columnar: each column stored together, so scans read only needed columns and compress well (similar values adjacent) — ideal for aggregations over few columns across many rows.

Q14. What is the write-ahead log (WAL) and why does it matter? hard

Changes are written to a sequential log and flushed to disk before the commit is acknowledged, giving durability without waiting for random data-page writes. On crash, the engine replays the log (redo) and rolls back uncommitted txns (undo). It's the foundation of both durability and crash recovery, and conceptually the same idea as a database changelog / CDC source.

PySpark & Apache Spark

The single most-tested DE skill. You must explain the execution model, the shuffle, joins, skew, and tuning — from memory, out loud. This is where mid-level candidates win or lose.

Practical theory you must be able to say out loud

Why Spark exists

Data too big for one machine → distribute it across a cluster and move compute to the data. Spark is a distributed engine that partitions data across executors and runs transformations in parallel, in memory where possible. It replaced MapReduce by keeping intermediate data in memory and using a smart optimizer.

Architecture — who runs what

Driver: runs your main, builds the DAG, schedules tasks, holds the SparkSession. Cluster manager (YARN/K8s/Standalone): allocates resources. Executors: JVM processes on worker nodes that run tasks and cache data. One task processes one partition. Jobs split into stages at shuffle boundaries; stages split into tasks.

Lazy evaluation: transformations vs actions

Transformations (select, filter, join, groupBy, withColumn) are lazy — they build a logical plan, nothing runs. An action (count, collect, write, show) triggers execution. Laziness lets Catalyst optimize the whole chain (pushdowns, pruning, reordering) before running. Narrow transformations (map, filter) need no data movement; wide transformations (groupBy, join, distinct) require a shuffle.

The shuffle — the thing to understand deeply

A shuffle repartitions data across the network so rows with the same key land on the same executor (needed for join/groupBy). It's expensive: disk spill + serialization + network + a stage boundary. Minimize shuffles: filter early, broadcast small tables, pre-partition, avoid unnecessary distinct/repartition. Shuffle count and size are the first thing to look at when tuning.

Joins & the killer: data skew

Spark join strategies: broadcast hash join (ship a small table to every executor — no shuffle, fastest when one side is small), sort-merge join (default for two large tables — both shuffled and sorted by key), and shuffle hash join. Skew = one key has far more rows than others, so one task runs forever while others finish. Fixes: enable AQE skew join handling, salt the hot key (add a random suffix, join, aggregate), or broadcast if one side is small.

Catalyst & Tungsten, AQE

Catalyst is the query optimizer (logical → optimized logical → physical plan). Tungsten is the execution engine (off-heap memory, code generation). AQE (Adaptive Query Execution) re-optimizes at runtime using actual stats: coalesces shuffle partitions, converts sort-merge to broadcast when a side turns out small, and splits skewed partitions. Turn it on (spark.sql.adaptive.enabled=true).

Partitioning & caching

repartition(n) does a full shuffle to n partitions (use to increase parallelism or before a big write); coalesce(n) merges partitions without a full shuffle (use to reduce partitions cheaply, e.g. avoid small files). Cache/persist a DataFrame only if reused multiple times (it's not free — it uses memory). Target ~128–200MB per partition; too many tiny partitions = scheduling overhead, too few = no parallelism and OOM.

Say-it-cold checklist: driver/executor/task/partition · lazy eval & transformations vs actions · narrow vs wide · what a shuffle is & how to reduce it · broadcast vs sort-merge join · data skew & salting · AQE · repartition vs coalesce · cache when reused · small-files problem · partition sizing · Catalyst/Tungsten.
Anti-patterns interviewers probe: collect() on a big DataFrame (OOMs the driver) · Python UDFs when a built-in exists (serialization cost, no Catalyst optimization — use pyspark.sql.functions or pandas/Arrow UDFs) · calling count() repeatedly · not filtering before a join · writing thousands of tiny files.

Reading the Spark UI (30-second version)

Jobs → Stages → Tasks. Look for: a stage with one straggler task (skew), heavy Shuffle Read/Write (too much data movement), high spill (memory pressure — increase partitions or memory), and long GC time. The DAG visualization shows shuffle boundaries. This is a very common "have you actually used Spark" question.

Interview Q&A (25)

Q1. Transformations vs actions? basic

Transformations (select, filter, join, groupBy) are lazy — they build a plan and return a new DataFrame without executing. Actions (count, collect, show, write) trigger the actual computation. Laziness lets Catalyst optimize the whole chain before running.

Q2. Narrow vs wide transformation? medium

Narrow: each output partition depends on one input partition — no data movement (map, filter, union). Wide: output partitions depend on many input partitions — requires a shuffle (groupBy, join, distinct, repartition). Wide transformations create stage boundaries and are where cost lives.

Q3. What is a shuffle and why is it expensive? medium

A shuffle redistributes data across executors so rows with the same key are colocated (for join/groupBy). It costs disk spill, serialization, and network transfer, and forces a new stage. Reducing shuffles — filter early, broadcast small tables, pre-partition — is the core of Spark tuning.

Q4. Explain the driver, executors, tasks, and partitions. basic

The driver runs your program, builds the DAG, and schedules work. Executors are JVM processes on workers that run tasks and cache data. A partition is a chunk of the data; one task processes one partition. Parallelism ≈ number of partitions running across executor cores.

Q5. Broadcast join vs sort-merge join — when each? hard

Broadcast: when one table is small enough to fit in each executor's memory (default threshold ~10MB, tunable) — ship it everywhere, no shuffle, very fast. Sort-merge: default for two large tables — both are shuffled and sorted by the join key. Force a broadcast with broadcast(df) when you know a side is small; AQE can switch to broadcast at runtime.

Q6. What is data skew and how do you fix it? hard

Skew is when a few keys hold most of the rows, so one task runs far longer than the rest (visible as a straggler in the UI). Fixes: enable AQE skew handling, salt the hot key (append a random number to split it, join to a replicated dimension, then aggregate), broadcast the small side, or filter/handle the hot key separately.

Q7. repartition vs coalesce? medium

repartition(n) does a full shuffle to exactly n partitions (can increase or decrease, balanced) — use before a wide op or big write. coalesce(n) merges existing partitions without a full shuffle (only decreases, may be unbalanced) — use to cut partition count cheaply and avoid small files.

Q8. When should you cache/persist? medium

Only when a DataFrame is reused across multiple actions (e.g., feeding several outputs or iterative algorithms). Caching costs memory and can evict other data, so caching a once-used DataFrame just wastes resources. Unpersist when done. Choose storage level (MEMORY_AND_DISK) based on size.

Q9. Why avoid Python UDFs, and what are the alternatives? hard

Python UDFs serialize each row between the JVM and a Python worker and are opaque to Catalyst (no pushdown/optimization), so they're slow. Prefer built-in pyspark.sql.functions; if you must, use pandas/vectorized (Arrow) UDFs which batch rows and are far faster. Never do row-at-a-time Python when a native function exists.

Q10. RDD vs DataFrame vs Dataset? medium

RDD: low-level, typed, no optimizer — full control, but you optimize manually. DataFrame: high-level, columnar, optimized by Catalyst — the default for DE. Dataset (Scala/Java): typed DataFrame with compile-time safety. In PySpark you use DataFrames; drop to RDDs only for rare low-level needs.

Q11. What is lazy evaluation good for? medium

It lets Spark see the entire transformation chain before executing, so Catalyst can push filters down to the scan, prune columns, reorder joins, and combine steps — producing a far more efficient physical plan than running each step eagerly. It also avoids computing unused results.

Q12. Why is collect() dangerous? basic

collect() pulls all partitions to the driver's memory. On a large DataFrame it OOMs the driver. Use it only on small results; otherwise write to storage, or use take()/limit() for a peek.

Q13. How do you handle the small-files problem? medium

Many tiny output files hurt read performance and metadata. Reduce output partitions with coalesce/repartition before write, size partitions to ~128MB+, use Delta OPTIMIZE (bin-packing) or periodic compaction, and partition the table by a sensible column (not too granular). Avoid partitioning by high-cardinality columns.

Q14. What is AQE and what does it do? hard

Adaptive Query Execution re-optimizes the plan at runtime using real shuffle statistics: it coalesces small shuffle partitions, switches sort-merge to broadcast when a side is actually small, and splits skewed partitions. Enable it (on by default in recent Spark) — it removes a lot of manual tuning.

Q15. How do you decide the number of partitions? hard

Aim for ~128–200MB per partition and at least as many partitions as total executor cores (often 2–4×) for good parallelism. Too few → underutilization and OOM/spill; too many → scheduling overhead and small files. spark.sql.shuffle.partitions (default 200) controls post-shuffle partitions; tune to data size, and let AQE coalesce.

Q16. What happens when a Spark task fails? medium

Spark retries the task (default up to 4 attempts) on another executor; because RDD/DataFrame lineage is deterministic, it recomputes the lost partition from its parents. If retries exhaust, the stage and job fail. This lineage-based recovery is why Spark is fault-tolerant without replicating intermediate data.

Q17. Explain jobs, stages, and tasks. medium

An action triggers a job. The job is split into stages at shuffle boundaries (each stage is a set of transformations with no shuffle between them). Each stage runs as parallel tasks, one per partition. The Spark UI shows this hierarchy — useful for spotting the expensive stage.

Q18. How do you read a partitioned Parquet/Delta table efficiently? medium

Filter on the partition column so Spark prunes partitions (partition pruning) and only reads matching files; select only needed columns (column pruning); rely on predicate pushdown into Parquet. Partition by a low/medium-cardinality column you filter on (e.g., date), not by something like user_id.

Q19. What is a broadcast variable vs a broadcast join? medium

A broadcast join is Spark auto-shipping a small table to executors for a join. A broadcast variable is you explicitly caching a read-only value (a lookup dict, config) on each executor so it isn't shipped with every task. Both avoid repeated data transfer.

Q20. Your Spark job is slow. Walk me through diagnosing it. hard

Open the Spark UI: find the longest stage. Check for (1) a straggler task → skew (salt/AQE); (2) huge shuffle read/write → reduce shuffles, broadcast, filter earlier; (3) high spill → more partitions or memory; (4) many tiny tasks → coalesce; (5) a Python UDF → replace with native; (6) reading too much → partition/column pruning. Fix the biggest bottleneck first, measure, repeat.

Q21. What is checkpointing and how does it differ from cache? hard

Cache/persist keeps data in memory/disk but preserves lineage (can be recomputed). Checkpoint writes the DataFrame to reliable storage and truncates lineage — used to break very long lineage chains (which slow planning and risk stack issues) and in streaming for fault-tolerant state/offset recovery.

Q22. How does Spark Structured Streaming work at a high level? hard

It treats a stream as an unbounded table, processing data in micro-batches (or continuous). It tracks offsets in a checkpoint for exactly-once, supports event-time windows with watermarks to handle late data and bound state, and writes with idempotent sinks. Same DataFrame API as batch, which is a big selling point.

Q23. What is a watermark in streaming? hard

A watermark tells Spark how late data can arrive ("up to 10 minutes late"), so it can finalize windows and drop state older than that. Without it, windowed aggregation state grows unbounded. It trades a little completeness (very late records dropped) for bounded memory.

Q24. How do you tune executor memory/cores? hard

Rule of thumb: ~4–5 cores per executor (more causes HDFS/IO contention), sized so partitions fit in executor memory with room for shuffle/overhead (leave ~10% for overhead). Avoid one giant executor (GC pain) or many tiny ones (overhead). Then watch spill/GC in the UI and adjust. Let AQE handle partition counts.

Q25. Explain how you'd write an idempotent Spark batch job. hard

Make re-running the same input produce the same output with no duplicates: process by partition/date, overwrite that partition (dynamic partition overwrite) or MERGE/upsert into Delta on a business key rather than blind append, derive deterministic outputs, and store the processed watermark. So a retry after failure is safe. This is exactly the reliability mindset from production support that interviewers want to hear.

Apache Kafka

Asked at a concept level: what it is, the data model, delivery semantics, and consumer groups. You use it at Jio — turn that into confident answers.

Practical theory you must be able to say out loud

What Kafka is

A distributed, durable, append-only commit log for events. Producers write events to topics; consumers read them. It decouples producers from consumers, buffers spikes, retains data for replay, and scales horizontally. Think of it as the "central nervous system" moving data between systems in real time.

The data model

A topic is split into partitions — the unit of parallelism and ordering. Each partition is an ordered, immutable log; every record has an offset (its position). Ordering is guaranteed only within a partition, not across a topic. Records with the same key go to the same partition (so per-key order holds). Partitions are replicated across brokers for durability; one replica is the leader, others are followers in the ISR (in-sync replicas).

Consumers & consumer groups

Consumers in a group share the work: each partition is consumed by exactly one consumer in the group, so max parallelism = number of partitions. Adding consumers beyond that leaves some idle. When consumers join/leave, a rebalance reassigns partitions. Consumers track progress by committing offsets. Different groups read the same topic independently (pub/sub).

Delivery semantics

At-most-once: commit offset before processing — may lose data. At-least-once: process then commit — may duplicate on retry (the common default; make consumers idempotent). Exactly-once: idempotent producer + transactions (read-process-write atomically) — real but has overhead. Durability tuning: producer acks=all + min.insync.replicas guarantees a write survives broker loss.

Say-it-cold checklist: topic/partition/offset · ordering only within a partition · keys → partition · replication/leader/ISR · consumer group = one consumer per partition · rebalancing · at-most/at-least/exactly-once · acks=all + min.insync.replicas · consumer lag · retention & replay · Kafka Connect & Schema Registry.
Trap: "How do you get more parallelism?" Add partitions — but you can't reduce partitions, and adding them breaks key→partition mapping for existing keys. Plan partition count up front.

Interview Q&A (15)

Q1. What is Kafka and when would you use it? basic

A distributed durable log for streaming events. Use it to decouple systems, ingest high-throughput real-time data, buffer bursts, feed multiple independent consumers, and enable replay. Common for event pipelines, CDC, metrics, and stream processing sources.

Q2. Topic, partition, offset? basic

Topic = a named stream of events. Partition = an ordered append-only shard of a topic (unit of parallelism/ordering). Offset = a record's sequential position within a partition, used by consumers to track what they've read.

Q3. Does Kafka guarantee ordering? medium

Only within a single partition, not across a topic. To keep related events ordered, give them the same key so they land in the same partition. If you need global ordering you're limited to one partition (no parallelism) — usually you design for per-key ordering instead.

Q4. How does a consumer group work? medium

Consumers in a group divide a topic's partitions so each partition is read by exactly one member — parallel consumption up to the partition count. Separate groups each get the full stream (pub/sub). Offsets are committed per group so each group tracks its own position.

Q5. What is a rebalance and why can it hurt? medium

When members join/leave or a consumer is deemed dead, Kafka reassigns partitions across the group. During a stop-the-world rebalance consumption pauses; frequent rebalances (from slow processing/timeouts) cause lag. Mitigate with cooperative/incremental rebalancing, proper session/poll timeouts, and keeping processing fast.

Q6. Explain replication, leader, and ISR. medium

Each partition has a replication factor; one replica is the leader (handles reads/writes), others replicate it. The ISR is the set of replicas caught up with the leader. If the leader fails, a new leader is elected from the ISR. min.insync.replicas defines how many must ack for a durable write.

Q7. At-least-once vs exactly-once — how do you achieve each? hard

At-least-once: process the message, then commit the offset — a crash between causes reprocessing (duplicates), so make consumers idempotent. Exactly-once: enable the idempotent producer plus Kafka transactions so consume-process-produce commits atomically with offsets, and downstream reads only committed messages. EOS has throughput cost; many pipelines choose at-least-once + idempotency instead.

Q8. What is consumer lag and why monitor it? medium

Lag = latest offset produced minus the offset the consumer has committed — how far behind real time you are. Growing lag means consumers can't keep up (scale consumers/partitions, speed up processing). It's the key health metric for a streaming pipeline.

Q9. How do producers decide the partition? medium

If a key is set, partition = hash(key) % partitions (same key → same partition, preserving order). No key → round-robin/sticky batching for balance. Choosing a good key controls both ordering and load distribution.

Q10. What does acks=0/1/all mean? medium

acks=0: fire-and-forget, fastest, can lose data. acks=1: leader acknowledges, lost if leader dies before replication. acks=all: leader waits for all in-sync replicas — strongest durability, higher latency. Pair acks=all with min.insync.replicas for real guarantees.

Q11. How is Kafka durable / how does retention work? medium

Messages are persisted to disk (append-only log segments) and replicated. Retention keeps data for a configured time or size (or compacted to keep the latest per key), independent of whether it's been consumed — enabling replay and multiple consumers. Consumers can reset offsets to reprocess.

Q12. What is Kafka Connect and Schema Registry? medium

Kafka Connect is a framework of ready-made source/sink connectors (DB CDC, S3, JDBC) to move data in/out without custom code. Schema Registry stores Avro/Protobuf/JSON schemas and enforces compatibility so producers and consumers evolve schemas safely. Together they standardize ingestion.

Q13. Kafka vs a traditional message queue (e.g., RabbitMQ)? medium

Queues typically delete a message once consumed and push to consumers; Kafka is a durable log where consumers pull and data is retained for replay by many independent groups. Kafka favors high-throughput streaming and reprocessing; classic queues favor per-message routing and complex delivery logic.

Q14. How would you handle a "poison" message that keeps failing? hard

Don't block the partition forever. Retry a bounded number of times with backoff, then route the bad record to a dead-letter topic (DLQ) with error metadata for later inspection, and commit the offset so processing continues. Alert on DLQ volume.

Q15. How do you scale a Kafka consumer application? medium

Add consumers to the group up to the partition count (parallelism ceiling), so provision enough partitions in advance. Beyond that, add partitions (carefully — it changes key mapping), speed up per-record processing, or fan out to multiple groups. Watch lag to know when to scale.

A managed orchestration/ELT service: you build pipelines of activities (copy data, run a notebook, call a stored proc) with triggers and monitoring. It moves data between many sources and coordinates transformation compute; it's the control plane, not the heavy compute.

Q3. Linked service vs dataset vs activity vs pipeline? medium

Q4. What is an Integration Runtime? medium

Q5. How do you design an incremental load in ADF? hard

Keep a watermark (last max modified timestamp/ID) in a control table. Pipeline: Lookup old watermark → Lookup new max from source → Copy rows where value > old and ≤ new → Stored Proc to update the watermark. It's incremental, idempotent, and restartable. For updates/deletes, land to staging and MERGE into the target.

Q6. Types of triggers? medium

Schedule (wall-clock recurring), Tumbling window (fixed, non-overlapping windows with dependency/backfill and retry — good for time-partitioned incremental loads), and Event-based (fires on blob created/deleted). Tumbling windows are favored for reliable incremental pipelines.

Q7. How do you manage secrets? basic

Q9. Copy activity: how do you make it fast? medium

Use parallel copy / partitioned reads (by a column or physical partitions), a staged copy through blob for some sinks, appropriate DIUs (data integration units), compression, and columnar formats (Parquet) at the sink. Filter at source to move less data.

Q10. How do you parameterize pipelines for reuse? medium

Use pipeline parameters and dataset parameters (table name, path, date), pass values from triggers or a metadata/control table, and drive a ForEach over a list of tables. This "metadata-driven" pattern loads many tables with one parameterized pipeline instead of one pipeline per table.

Q11. How do you handle failures and retries in ADF? medium

Set activity retry count/interval, use timeout, add failure paths (on-failure dependencies) to log/alert, and design idempotent loads so a rerun is safe. Monitor via the ADF monitoring UI/alerts. Tumbling-window triggers give built-in retry and backfill.

Q12. Mapping data flows vs a Databricks notebook — when each? medium

Mapping data flows are ADF's visual, Spark-backed transformations — good for simpler, code-light transforms. For complex logic, custom libraries, ML, or when the team already uses Spark, call a Databricks notebook. Choose by complexity and team skill; data flows still spin up Spark under the hood.

One pipeline scales to all tables and is easy to add to — exactly what production teams build.

Databricks & Delta Lake

You use it daily. Interviewers want the lakehouse concepts: Delta's ACID/time-travel, MERGE, OPTIMIZE/Z-ORDER, VACUUM, and the medallion pattern.

Practical theory you must be able to say out loud

What Delta Lake is

An open table format that sits on top of Parquet files in a lake and adds a transaction log (_delta_log). That log gives you ACID transactions, schema enforcement/evolution, time travel (query old versions), upserts/deletes (MERGE), and reliable concurrent reads/writes — turning a messy lake into a "lakehouse" with warehouse-like guarantees on cheap storage.

How the transaction log works

Each write creates a new JSON commit in _delta_log listing which Parquet files were added/removed. Readers reconstruct the current table from the log (not by listing the directory), which is what enables atomic commits, snapshot isolation, and time travel. Periodic checkpoints (Parquet) speed up log replay.

Key operations

MERGE: upsert — match on a key, update/insert/delete (the core of SCD and CDC). Time travel: VERSION AS OF / TIMESTAMP AS OF for audits, rollbacks, reproducibility. OPTIMIZE: bin-pack small files into larger ones (fixes the small-files problem); Z-ORDER: co-locate related data to improve data-skipping on filter columns. VACUUM: physically delete files no longer referenced (older than a retention window) to reclaim storage — but it removes the ability to time-travel past that point.

Medallion architecture on Databricks

Bronze (raw ingested Delta) → Silver (cleaned, deduped, conformed) → Gold (aggregated marts). Autoloader/Structured Streaming ingests incrementally; Delta MERGE handles updates; Unity Catalog governs access and lineage across the layers.

Say-it-cold checklist: Delta = Parquet + transaction log · ACID/time travel/schema enforcement · MERGE for upserts/CDC · OPTIMIZE + Z-ORDER (small files & data skipping) · VACUUM (reclaim, limits time travel) · medallion bronze/silver/gold · Autoloader · Unity Catalog · jobs vs interactive clusters · why lakehouse beats separate lake+warehouse.
Trap: running VACUUM with a short retention breaks time travel and can corrupt readers/streams mid-write — keep the default 7-day retention unless you understand the consequences.

Interview Q&A (16)

Q1. What problem does Delta Lake solve? basic

Plain lakes (Parquet on object storage) lack transactions, so concurrent writes, partial failures, and updates corrupt or duplicate data, and there's no schema enforcement or history. Delta adds a transaction log giving ACID, schema control, upserts, and time travel — reliability of a warehouse on the economics of a lake.

Q2. How does Delta provide ACID? medium

Through the ordered _delta_log: each commit atomically records file adds/removes, readers see a consistent snapshot (snapshot isolation), and optimistic concurrency control resolves conflicting writers. A write is all-or-nothing because it's a single log commit.

Q3. What is time travel and when do you use it? medium

Query a previous table version by version number or timestamp. Uses: audit/debug (what did the data look like yesterday), rollback a bad load, reproduce an ML training set, or compare versions. Limited by the retention window (before VACUUM removes old files).

Q4. Explain MERGE and a use case. medium

MERGE upserts: match target and source on a key, then WHEN MATCHED update/delete and WHEN NOT MATCHED insert — atomically. Core use cases: applying CDC changes, SCD Type 1/2 dimension maintenance, and idempotent loads (re-run without duplicating).

Q5. What do OPTIMIZE and Z-ORDER do? hard

OPTIMIZE compacts many small files into fewer large ones, cutting read overhead (the small-files problem). Z-ORDER (a multi-dimensional clustering) co-locates rows with similar values in the columns you filter on, improving data skipping via min/max stats so queries scan fewer files. Z-ORDER on high-selectivity filter columns.

Q6. What does VACUUM do and what's the risk? medium

It deletes data files no longer referenced by the log and older than the retention period, reclaiming storage. Risk: it removes the ability to time-travel before that point, and vacuuming too aggressively can break concurrent readers/streaming jobs relying on those files. Keep the default retention.

Q7. What is the medallion architecture? basic

Bronze (raw, immutable landing), Silver (cleaned/conformed/deduped), Gold (business aggregates/marts). Each layer is a Delta table, incrementally built and independently reprocessable, improving quality as data flows up.

Q8. What is Autoloader? medium

Databricks' incremental file ingestion (cloudFiles): it efficiently detects and loads new files from cloud storage (via notifications or listing), tracks what's been processed with a checkpoint (exactly-once), and handles schema inference/evolution. Ideal for continuously landing files into bronze.

Q9. Schema enforcement vs schema evolution? medium

Enforcement rejects writes whose schema doesn't match the table (prevents silent corruption). Evolution lets you intentionally add columns (e.g., mergeSchema) so the table adapts to new fields. Together they give safety by default with controlled change.

Q10. Job cluster vs all-purpose (interactive) cluster? medium

Interactive/all-purpose clusters are for development and shared notebooks (stay up, costlier). Job clusters are spun up for a scheduled job and torn down after — cheaper and isolated for production workloads. Use job clusters for scheduled pipelines.

Q11. What is Unity Catalog? medium

Databricks' centralized governance layer: a three-level namespace (catalog.schema.table), fine-grained access control, data lineage, auditing, and discovery across workspaces. It replaces per-workspace hive metastores with unified, governed metadata.

Q12. How do you apply CDC into a Delta table? hard

Land change records (insert/update/delete with a sequence/timestamp) to staging, deduplicate to the latest change per key, then MERGE into the target: update on matched updates, delete on matched deletes, insert on new. Use Delta Change Data Feed to emit downstream changes. It's idempotent and ordered by the sequence column.

Q13. Why is the lakehouse better than a separate lake + warehouse? medium

One copy of data with transactions, governance, and BI performance directly on the lake — no costly, drift-prone ETL to a separate warehouse, no duplicate storage, and both SQL analytics and ML/streaming on the same tables. Lower cost and less inconsistency.

Q14. How do you handle late/duplicate data in a streaming Delta pipeline? hard

Use a checkpoint for exactly-once offset tracking, watermarks to bound state and handle late events, and MERGE/dedup on an event id so reprocessing doesn't duplicate. Idempotent MERGE plus deterministic keys means a replay is safe.

Q15. Delta vs Iceberg vs Hudi — one line? medium

All are open transactional table formats adding ACID/time-travel/upserts to lakes. Delta is Databricks-native and most common there; Iceberg has strong engine-neutral adoption (Snowflake, Trino, Flink) and hidden partitioning; Hudi excels at streaming upserts/CDC. The concepts transfer across all three.

Q16. How do you optimize a slow Delta query? hard

Ensure files are compacted (OPTIMIZE), Z-ORDER/partition on filter columns for data skipping, prune partitions and columns in the query, avoid tiny files from streaming (auto-compaction), keep statistics current, and cache hot tables. Check the Spark UI for the actual bottleneck (skew/shuffle) rather than guessing.

Snowflake

Cloud data warehouse. Know the compute/storage separation, micro-partitions, virtual warehouses, and the cost model — that's what gets asked.

Practical theory you must be able to say out loud

The defining idea: separation of storage and compute

Snowflake stores data centrally (compressed, columnar, in cloud object storage) and runs queries on independent virtual warehouses (compute clusters) that you size and start/stop separately. Multiple warehouses can query the same data at once without contention, and you pay for storage and compute independently. This elasticity is the whole pitch.

Micro-partitions & pruning

Snowflake automatically stores data in immutable micro-partitions (~50–500MB compressed), each with metadata (min/max per column). Queries prune partitions using that metadata so they scan only relevant data — no manual indexes. Clustering keys help pruning on very large tables by keeping related data together.

Features you should name

Time Travel (query/restore data up to N days back), Zero-copy cloning (instant metadata-only copies of tables/DBs for dev/test), Streams (CDC — track table changes) + Tasks (scheduled SQL) for pipelines, result cache (repeat queries free), and multi-cluster warehouses for concurrency scaling.

Interview Q&A (12)

Q1. What makes Snowflake's architecture different? basic

It separates storage from compute (and from cloud services). Data lives once in central object storage; independent virtual warehouses compute over it. You scale compute up/out or spin up isolated warehouses without moving data, and pay for each separately.

Q2. What is a virtual warehouse? basic

A named, resizable compute cluster (XS→4XL) that executes queries. You can auto-suspend it when idle and auto-resume on demand to control cost, and run separate warehouses per team/workload so they don't contend. Bigger size = more parallelism per query; multi-cluster = more concurrency.

Q3. What are micro-partitions and how do they help? medium

Snowflake auto-divides tables into small immutable columnar chunks with per-column min/max metadata. Queries use that metadata to skip (prune) partitions that can't match the filter, scanning far less data — automatically, without user-managed indexes.

Q4. What is a clustering key and when do you need one? medium

An explicit ordering hint for very large tables so related rows stay in the same micro-partitions, improving pruning on that column. Only needed for large tables with queries filtering on a specific column where natural clustering has degraded; Snowflake can auto-recluster. Over-clustering costs credits.

Q5. How does Snowflake's cost model work? medium

You pay for storage (compressed, per TB) and compute (credits per warehouse-second while running). Control cost with auto-suspend, right-sizing warehouses, avoiding always-on compute, leveraging result cache, and reducing scanned data via pruning/clustering. Idle warehouses that don't suspend are the classic cost leak.

Q6. What is Time Travel and Fail-safe? medium

Time Travel lets you query or restore data as of a point in the past (up to 1 day standard, up to 90 on Enterprise) — great for undoing mistakes. Fail-safe is an additional 7-day Snowflake-managed recovery window after Time Travel, for disaster recovery only (not self-serve).

Q7. Explain zero-copy cloning. medium

CLONE creates an instant copy of a table/schema/database by copying metadata pointers, not data — no extra storage until the clone diverges (copy-on-write). Perfect for spinning up dev/test/QA environments from production instantly and cheaply.

Q8. Streams and Tasks — what are they for? hard

A Stream tracks row-level changes (CDC) on a table since last consumed. A Task runs scheduled SQL (or a DAG of tasks). Combine them to build native ELT: a stream captures new/changed rows, a task periodically MERGEs them downstream — an in-warehouse incremental pipeline.

Q9. How does Snowflake load data (COPY / Snowpipe)? medium

Bulk load with COPY INTO from a stage (internal or external cloud storage), typically compressed Parquet/CSV. Snowpipe does continuous, near-real-time micro-batch loading triggered by file-arrival notifications. Use COPY for scheduled batches, Snowpipe for streaming file ingestion.

Q10. What is the result cache? basic

Snowflake caches query results for 24 hours; an identical query on unchanged data returns instantly with no compute cost. There are also warehouse-local data caches. Good to mention when asked how repeated dashboards stay cheap/fast.

Q11. How do you handle semi-structured data (JSON) in Snowflake? medium

Load it into a VARIANT column and query with path notation (col:field.subfield) and functions like FLATTEN for arrays. Snowflake stores VARIANT columnar-ly and prunes on it, so you can keep raw JSON and shred it on read or into typed columns.

Q12. Snowflake vs Databricks — when would you pick each? hard

Snowflake: SQL-first analytics/BI, easy elasticity, minimal ops, great for warehousing and semi-structured data. Databricks: Spark-based data engineering, ML, streaming, and open lakehouse (Delta) with code-heavy transforms. Many orgs use both — Databricks for engineering/ML, Snowflake for serving BI — and the line is blurring as each adds the other's features.

Apache Airflow

Orchestration. Know the architecture, the DAG model, the (confusing) logical date, idempotency, and how retries/backfills work.

Practical theory you must be able to say out loud

Why orchestration exists

Cron can't express dependencies, retries, backfills, SLAs, or visibility. Airflow runs workflows as DAGs (directed acyclic graphs) of tasks with dependencies, scheduling, retries, alerting, and a UI — so complex, dependent pipelines run reliably and are observable.

Architecture

Scheduler: parses DAGs, decides what to run when, and queues tasks. Metadata DB: stores DAG/task state (the source of truth). Executor: how tasks run (Local, Celery, Kubernetes). Workers: execute tasks. Webserver: the UI. A task instance = one run of a task for one logical date.

The logical date (execution_date) — the classic confusion

A scheduled run's logical date is the start of the interval it covers, and the DAG runs at the end of that interval. So a daily DAG for data-date 2026-07-12 actually executes just after midnight on 2026-07-13. You use the logical date to parameterize which data partition to process — never datetime.now(). This is what makes runs deterministic and backfillable.

Idempotency & retries

Tasks should be idempotent: rerunning for the same logical date produces the same result (overwrite/MERGE the partition, don't blind-append). Then retries and backfills are safe. Airflow gives per-task retries with delay, sensors (wait for a file/partition — use deferrable/reschedule mode to free workers), SLAs, and trigger rules. Keep tasks small and atomic.

Say-it-cold checklist: DAG/task/operator · scheduler/metadata DB/executor/worker · logical date = interval start, runs at interval end · use logical date not now() · idempotent tasks → safe retries/backfills · XCom for small data (not big payloads) · sensors (deferrable) · retries/SLAs/trigger rules · Celery vs Kubernetes executor.
Traps: passing large data through XCom (it's for small values — pass paths/pointers instead) · using datetime.now() in tasks (breaks backfills and idempotency) · top-level heavy code in the DAG file (the scheduler parses it constantly).

Interview Q&A (14)

Q1. What is a DAG? basic

A Directed Acyclic Graph of tasks with dependencies — no cycles, so there's a clear order. It defines what runs, in what sequence, and how often. Each execution for a given interval is a DAG run made of task instances.

Q2. Describe Airflow's architecture. medium

Scheduler decides and queues tasks; the metadata database stores all state; the executor (Local/Celery/Kubernetes) determines how/where tasks run; workers execute them; the webserver serves the UI. The scheduler + metadata DB are the brain; workers are the muscle.

Q3. Explain the logical/execution date. hard

It's the start of the data interval the run covers, and the DAG fires at the interval's end. A daily job for 2026-07-12 runs after midnight on the 13th. Parameterize the data partition by this date (not now()) so runs are deterministic and backfills reprocess the correct historical partitions.

Q4. Why must tasks be idempotent? medium

Because Airflow retries failed tasks and you backfill history — rerunning must not duplicate or corrupt data. Achieve it by overwriting/MERGE-ing the target partition for the logical date rather than appending, and by deriving deterministic outputs. Idempotency is what makes retries/backfills safe.

Q5. What is XCom and its limits? medium

Cross-communication: a way to pass small values between tasks via the metadata DB. It's not for large data — passing big payloads bloats the DB and is slow. Pass file paths, table names, or row counts, and let downstream tasks read the actual data from storage.

Q6. What are sensors and how do you use them efficiently? medium

Sensors wait for a condition (a file lands, a partition exists, an external job finishes). Naive sensors occupy a worker slot while polling; use reschedule mode or deferrable (async) sensors so they release the worker between checks — critical at scale to avoid worker starvation.

Q7. What is a backfill? medium

Running a DAG for past dates it hasn't run (or rerunning history after a fix). Because tasks are parameterized by logical date and idempotent, Airflow can reprocess each historical interval correctly. Control concurrency so a big backfill doesn't overwhelm sources.

Q8. How do retries and alerting work? basic

Set retries and retry_delay per task; Airflow re-runs on failure with backoff. Configure on-failure callbacks / email / Slack alerts and SLAs for late runs. Combine with idempotency so retries are safe.

Q9. Celery vs Kubernetes executor? medium

Celery uses a pool of long-running workers via a message broker — good throughput, but workers are always on. Kubernetes executor launches a pod per task — isolated, elastic, no idle workers, better dependency isolation, at the cost of pod startup latency. Pick K8s for isolation/scale-to-zero, Celery for steady high volume.

Q10. How do you avoid heavy code at the top level of a DAG file? hard

The scheduler parses every DAG file frequently, so top-level DB calls/API queries/imports slow the whole scheduler. Keep top-level code light (just structure); do heavy work inside task callables, use dynamic task mapping carefully, and cache/limit external calls during parsing.

Q11. What are trigger rules? medium

They control when a task runs based on upstream states — default all_success, plus all_done, one_success, none_failed, etc. Used for cleanup tasks that must run regardless of failure, or branching/fan-in logic.

Q12. How do you handle dependencies between DAGs? medium

Use an ExternalTaskSensor (wait for another DAG's task), the TriggerDagRunOperator (fire a downstream DAG), or Airflow Datasets (data-aware scheduling that triggers a DAG when an upstream dataset updates). Datasets are the cleaner modern approach for producer/consumer pipelines.

Q13. Your DAG runs but processes the wrong day's data — why? hard

Almost always using datetime.now() instead of the logical date, or misunderstanding that the run for interval-start date executes at interval-end. Fix by templating the data path/partition with the logical date ({{ ds }}) so each run targets the correct partition and backfills work.

Q14. How would you orchestrate a daily ingest → transform → publish pipeline? hard

A DAG with a schedule; a sensor/Dataset waits for source readiness; an extract task loads the logical-date partition to bronze (idempotent overwrite); a Databricks/Spark task transforms to silver/gold; a DQ-check task validates row counts/nulls and fails the run if bad; a publish task; alerts on failure/SLA. Each task idempotent and parameterized by logical date.

Data Quality & Reliability

Your Jio production-support experience IS this topic. It's senior-signal — reconciliation, RCA, idempotency, SLAs. Lead with real incidents.

Practical theory you must be able to say out loud

The quality dimensions

Completeness (nothing missing — row counts, no unexpected NULLs), Accuracy (values are correct vs source of truth), Consistency (agrees across systems — reconciliation), Uniqueness (no duplicates on the key), Validity (conforms to type/format/range/enum), Timeliness (arrives within SLA), Referential integrity (FKs resolve). Frame any DQ answer around these.

Where checks live

Push checks left. Schema/contract at ingestion (reject or quarantine bad structure), row-level validation in transform (drop/flag/quarantine bad rows), aggregate checks after load (counts, sums, distributions, reconciliation to source), and freshness/volume anomaly monitoring. Tools: Great Expectations / dbt tests / custom assertions. Fail loud on critical checks; warn on soft ones.

Reconciliation

Proving the pipeline neither lost nor invented data: compare source vs target on counts and control totals (sum of a key measure), row-hash comparisons for detail, and balance checks (e.g., debits = credits). Any mismatch triggers RCA. This is exactly what production support does daily — and it's gold in interviews.

Reliability: idempotency & correct reprocessing

Design so any run can be safely retried: process by partition/logical date, overwrite or MERGE (not blind append), use deterministic transforms, track a watermark, and make writes atomic (Delta/temp-then-swap). Then failures are recoverable without duplicates or gaps.

Interview Q&A (18)

Q1. What are the dimensions of data quality? basic

Completeness, accuracy, consistency, uniqueness, validity, timeliness, and referential integrity. A strong answer names them and gives a concrete check for each (row counts for completeness, dedup for uniqueness, range checks for validity, reconciliation for consistency).

Q2. How do you validate a pipeline's output? medium

Layered checks: schema/contract at ingest, row-level validation in transform, and aggregate checks after load (row counts vs source, null rates, distinct-key counts, control-total sums, referential integrity). Automate them as pipeline steps that fail the run on critical violations and alert.

Q3. What is reconciliation and how do you do it? medium

Confirming the target matches the source: compare record counts, compare control totals (sum of a key amount), and for detail use row-level hashes or key set differences. Mismatches → investigate (RCA). It proves no rows were dropped, duplicated, or altered in transit.

Q4. A daily count is suddenly 30% lower. Walk me through RCA. hard

Confirm it's real (check the DQ metric, not a display bug). Scope it: which source/partition/step dropped? Compare source vs staging vs target counts to localize where volume was lost. Check upstream (source outage, late data, schema change), the pipeline (a filter/join dropping rows, a failed partition, dedup over-collapsing), and time window (holiday/seasonality). Fix root cause, backfill the affected partition idempotently, add a check to catch it earlier, and communicate impact.

Q5. What does idempotency mean and why does it matter? medium

Running the same job for the same input/date yields the same result with no duplicates. It matters because failures and retries are inevitable — idempotent jobs are safely restartable and backfillable. Achieve it by overwriting/MERGE-ing the target partition keyed by logical date instead of appending.

Q6. How do you handle a bad/duplicate record mid-pipeline? medium

Decide by severity: reject and quarantine (route to an error table with the reason) for recoverable bad data so the good data flows, drop with logging for noise, or fail fast for systemic corruption. Never silently discard — quarantine keeps an audit trail and lets you reprocess after fixing.

Q7. What is a data contract? medium

An agreed, enforced specification between a data producer and consumer — schema, types, semantics, freshness, allowed nulls, and change policy. Enforcing it at ingestion catches breaking upstream changes before they corrupt downstream, shifting the failure left to the producer.

Q8. How do you detect data freshness/volume anomalies? medium

Track per-run metrics — arrival time vs SLA, row counts, null rates, distinct counts — and alert on deviations from expected ranges or historical baselines (static thresholds or anomaly detection). Freshness and volume are the earliest signals something upstream broke.

Q9. Fail-fast vs quarantine — how do you choose? medium

Fail-fast when a violation means the whole output is untrustworthy (schema break, control-total mismatch) — stop before publishing bad data. Quarantine when most data is good and a minority is bad — isolate the bad rows, let the good ones proceed, and reprocess the quarantine later. Match the blast radius.

Q10. What is data lineage and why is it useful? medium

The traceable path of data from source through transformations to outputs. It powers impact analysis (what breaks if this source changes), debugging (where did this wrong number come from), and governance/compliance. Tools capture it automatically (Unity Catalog, OpenLineage).

Q11. How do you ensure exactly-once semantics end to end? hard

Combine idempotent writes (MERGE/upsert on a business key, or overwrite partition) with deduplication on a unique event id and checkpointed offsets. True end-to-end exactly-once usually = at-least-once delivery + idempotent, deduplicating sinks, rather than distributed transactions everywhere.

Q12. How would you build monitoring for a critical pipeline? hard

Instrument each stage: success/failure and duration, input/output row counts, DQ check results, freshness vs SLA, and reconciliation status. Dashboard the metrics, alert on failures/SLA breach/anomalies with clear ownership and runbooks, and log enough context for fast RCA. Distinguish page-worthy from FYI alerts to avoid fatigue.

Q13. A downstream schema change broke your pipeline. How do you prevent recurrence? hard

Enforce a data contract / schema validation at ingest so breaking changes are caught and quarantined immediately with a clear error, use schema evolution where additive changes are expected, add a schema-drift alert, and coordinate change management with the producer. Shift the failure to the boundary, not deep in transforms.

Q14. What's the difference between a warning and a hard-fail check? basic

A hard-fail check stops the pipeline and blocks publishing when violated (critical: schema, control totals, PK uniqueness). A warning logs/alerts but lets data flow (soft: minor null-rate drift). Classify each check by business impact so you neither publish bad data nor cry wolf.

Q15. How do you backfill after fixing a bug without double counting? hard

Reprocess only the affected logical-date partitions with an idempotent job that overwrites/MERGEs those partitions, so reruns replace rather than append. Validate counts against source after, and confirm downstream aggregates recomputed. The idempotent, partition-scoped design is what makes this safe.

Q16. Tell me about a production incident you handled. medium

Use STAR with your real Jio experience: the alert/symptom, how you triaged and localized it (logs, counts, reconciliation), the root cause, the fix and safe backfill, and the preventive measure (new check/alert). Emphasize impact minimized and recurrence prevented — that's the senior signal.

Q17. How do you test data pipelines before production? medium

Unit-test transform functions on small fixtures (including edge cases), run integration tests on sample data end-to-end, assert output DQ expectations, test idempotency (run twice → same result), and validate against a known-good baseline. Use dev/staging environments (zero-copy clones help) before promoting.

Q18. What SLAs/SLOs would you set for a pipeline and why? hard

Freshness (data available by a time), completeness (≥ expected volume), and correctness (reconciliation passes) tied to business need. Define them with consumers, monitor and alert on breaches, and track error budgets so you balance new work vs reliability. Concrete, measurable, owned — not vague "it should work."

Just-enough theory

Git: distributed version control. Know branch → commit → push → pull request → merge, resolving conflicts, and merge vs rebase (rebase rewrites history for a linear log; don't rebase shared branches). A trunk-based or feature-branch workflow with PR review is standard.

CI/CD: CI runs tests/linting on every push (catch breakage early); CD deploys automatically after checks pass. For data: also test transformations, validate schemas, and deploy DAGs/notebooks/dbt models through environments (dev → staging → prod) with the same code.

Interview Q&A (10)

Q1. merge vs rebase? medium

Merge combines branches preserving history with a merge commit. Rebase replays your commits on top of another branch for a linear history but rewrites commit hashes — never rebase a branch others have pulled. Merge for shared branches; rebase locally to tidy before a PR.

Q2. How do you resolve a merge conflict? basic

Git marks conflicting regions; you edit to the correct combined result, remove the markers, stage the file, and complete the merge/rebase. Then run tests. Prevent conflicts by pulling often and keeping branches short-lived.

Q3. What problem does Docker solve for DE? basic

Environment reproducibility — the exact Python/Spark/library versions ship with the code, so a job runs identically on a laptop, CI, and production. It removes dependency drift and makes deployments predictable.

Q4. Image vs container? basic

An image is the immutable, built template (layers of filesystem + config). A container is a running instance of an image. You build once, run many containers from the same image.

Q5. What is CI/CD and why does it matter for pipelines? medium

CI automatically tests/lints/validates code (including transformation and schema tests) on each push; CD deploys through environments once checks pass. For data teams it prevents broken DAGs/models reaching prod and makes releases repeatable and auditable.

Q6. How would you deploy Airflow DAGs safely? medium

Version DAGs in Git, run CI (lint, import tests, unit tests) on PRs, and deploy to dev/staging before prod via CD (sync to the DAGs folder or bake into an image). Test that DAGs parse and tasks are idempotent before promotion.

Q7. What is a .gitignore and what should never be committed? basic

It excludes files from version control (build artifacts, venvs, data files). Never commit secrets/credentials, large data, or environment-specific config — use env vars/secret managers instead. Committed secrets must be rotated immediately.

Q8. How do you manage environments (dev/staging/prod)? medium

Same code, environment-specific config (connections, paths, credentials) injected via variables/secret scopes. Promote artifacts through environments with CI/CD; use isolated storage/warehouses per env. Zero-copy clones or sampled data give realistic non-prod testing.

Q9. What is a feature branch workflow? basic

Create a short-lived branch per change, commit there, open a pull request for review and CI, then merge to main. Keeps main deployable, enables review, and isolates work. Trunk-based development keeps branches very short to reduce merge pain.

Q10. How do you handle secrets in code/deployments? medium

Never in code or Git. Use a secret manager/vault (Azure Key Vault, Databricks secret scopes) or environment variables injected at runtime, with access via managed identity/RBAC and rotation. Reference secrets by name so code stays portable across environments.

Data Engineering System Design

The senior-round differentiator. You don't need a perfect answer — you need a structured approach and the right trade-offs spoken aloud.

The framework (use this on every prompt)

1. Clarify requirements. Ask: batch or streaming? Data volume & velocity? Latency/freshness SLA? Query patterns (who consumes, how)? Consistency needs? Budget? Never start designing before scoping.

2. Back-of-envelope math. Events/day → per second; bytes/event → daily & yearly storage; peak vs average. This sizes everything and shows senior thinking.

3. Draw the pipeline stages. Ingestion → Storage → Processing → Serving, plus orchestration and observability across all of it.

4. Make deliberate choices at each stage (below) and state the trade-off.

5. Address cross-cutting concerns: data quality, idempotency/exactly-once, schema evolution, partitioning, cost, security/governance, monitoring, failure & backfill.

6. Discuss bottlenecks & scaling. Where does it break at 10×? Skew, hot partitions, small files, shuffle, serving load.

The design space at each stage

Ingestion: batch (scheduled pulls, files) vs streaming (Kafka/Kinesis) vs CDC (Debezium/Snowflake streams). Push vs pull. Choose by latency SLA.

Storage: lake (Parquet/Delta on object storage) vs warehouse (Snowflake) vs both (lakehouse). Partition by a low-cardinality, frequently-filtered column (usually date). Columnar for analytics.

Processing: batch (Spark/SQL) vs stream (Structured Streaming/Flink). ELT (transform in warehouse) is the modern default; ETL when you must transform before load. Medallion layering.

Serving: BI/warehouse tables, a low-latency store (key-value/OLAP like ClickHouse/Druid) for dashboards, an API, or a feature store for ML. Match to query pattern.

Three worked designs (be ready to sketch these)

A. Daily batch analytics warehouse. Sources → ADF/Airflow orchestrated incremental extract (watermark) → bronze Delta on ADLS → Spark clean/conform → silver → gold star-schema marts → Snowflake/Power BI. DQ checks + reconciliation at each layer; idempotent partition overwrites; backfillable.

B. Real-time event pipeline. App → Kafka (partitioned by key) → Structured Streaming (windowed aggregation with watermark, checkpointed) → Delta (MERGE for upserts) → served to a low-latency OLAP store + dashboards. At-least-once + idempotent sink for exactly-once effect. Monitor consumer lag.

C. CDC replication. Source DB → Debezium/native CDC → Kafka → consumer dedups & orders by sequence → MERGE into Delta/warehouse (SCD as needed) → downstream marts. Handles inserts/updates/deletes; idempotent and replayable from offsets.

Golden phrases to say: "What's the latency SLA?" · "Let me size it first" · "I'd partition by date to enable pruning" · "I'll make it idempotent so retries/backfills are safe" · "I'd add a reconciliation check here" · "This will skew on hot keys, so I'd salt or broadcast" · "At 10× the bottleneck becomes X."

Interview Q&A (10)

Q1. Design a pipeline to ingest 1B events/day for analytics. hard

Clarify latency (near-real-time?) and consumers. Math: ~11.6k events/s average, plan for ~3–5× peak. Kafka (well-partitioned) for ingest → Structured Streaming or micro-batch → partitioned Delta (bronze→silver→gold) on object storage → serve gold to warehouse/OLAP. Address partitioning by event date, small-files compaction, idempotent writes, schema registry, watermarks for late data, and consumer-lag monitoring. Discuss scaling: add partitions/executors, handle skew.

Q2. Batch vs streaming — how do you decide? medium

Driven by the freshness SLA and cost. If consumers need data within seconds/minutes and value real-time, stream; if hourly/daily is fine, batch is simpler, cheaper, and easier to reason about. Many systems are hybrid (streaming ingest, batch heavy transforms — Lambda; or one streaming path — Kappa).

Q3. How would you design an idempotent ingestion layer? hard

Key each record with a stable id, process by logical-date partition, and write via MERGE/upsert or partition overwrite so reruns replace rather than duplicate. Track processed offsets/watermarks in a checkpoint/control table. Result: any retry or backfill is safe and produces the same output.

Q4. How do you handle late-arriving data? hard

In streaming: watermarks define how late is accepted and bound window state; very late data is dropped or routed to a correction path. In batch: reprocess affected partitions on a lookback window (e.g., reprocess last 3 days each run) with idempotent overwrites so corrections land without duplicates.

Q5. Design a data quality framework for a platform. hard

Contract/schema checks at ingest; row-level validations in transform (quarantine bad rows); aggregate checks after load (counts, nulls, control totals, reconciliation to source); freshness/volume anomaly monitoring; classify checks as hard-fail vs warn; central metrics + alerting with ownership/runbooks; lineage for impact analysis. Automated as pipeline steps, not manual.

Q6. How do you choose a partitioning strategy? medium

Partition by a low/medium-cardinality column that queries filter on (usually date) to enable pruning, targeting reasonable file sizes (~128MB+). Avoid high-cardinality partitions (user_id) — they create millions of tiny files. For skewed access, add sub-partitioning or clustering/Z-ordering.

Q7. Lambda vs Kappa architecture? hard

Lambda runs parallel batch and speed layers then merges — accurate but two codebases to maintain. Kappa uses a single streaming pipeline and reprocesses by replaying the log — simpler, one codebase, favored now that streaming engines are mature. Choose Kappa unless you have a strong batch-only requirement.

Q8. How would you make a pipeline cost-efficient? medium

Scan less (partition/column pruning, compaction, columnar formats), right-size and auto-suspend compute, prefer incremental over full loads, cache/materialize reused results, use spot/ephemeral job clusters, and archive cold data to cheaper storage tiers. Measure bytes scanned and compute-hours, not vibes.

Q9. Design a CDC pipeline from an OLTP DB to the warehouse. hard

Capture changes via log-based CDC (Debezium/native) → Kafka topic per table → consumer orders by the change sequence and dedups → MERGE into staging/target (apply inserts/updates/deletes, SCD2 where history is needed) → build marts. It's low-impact on source, near-real-time, idempotent, and replayable from offsets. Handle schema changes via a registry/contract.

Q10. What are the failure modes and how do you make it resilient? hard

Source outage/late data (retries, sensors, lookback reprocessing), bad data (contracts, quarantine), skew/hot partitions (salting/broadcast), small files (compaction), compute failure (idempotent retries, checkpoints), and schema drift (enforcement + alerts). Add monitoring, alerting with runbooks, and tested backfill so recovery is routine, not heroic.
